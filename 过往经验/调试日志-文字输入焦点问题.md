# WordFlow 文字输入焦点问题 - 调试日志

## 问题概述

**现象**：语音识别完成后，文字只能显示在 WordFlow 自己的编辑器中，无法输入到外部窗口（记事本、Word、WPS、浏览器等）。

**影响**：这是语音输入软件的核心功能，用户需要的是"说话→文字出现在光标处"，而不仅仅是"说话→软件里看到文字"。

## 调试时间线

### 2026-02-11 尝试方案 1：SendInput Unicode 输入

**实现思路**：
- 使用 Windows API `SendInput` 直接发送 Unicode 字符
- 每个字符转成键盘事件序列

**代码**：
```csharp
[DllImport("user32.dll")]
static extern uint SendInput(uint nInputs, INPUT[] pInputs, int cbSize);

// 发送 Unicode 字符
input.ki.wScan = c;
input.ki.dwFlags = KEYEVENTF_UNICODE;
```

**结果**：❌ 失败
- 日志显示发送成功
- 文字仍出现在 WordFlow 编辑器

**原因分析**：`SendInput` 只能发送到当前前台窗口，焦点恢复不彻底。

## 可能原因汇总

### 1. Windows UIPI（用户界面特权隔离）
Windows Vista+ 引入的安全机制，阻止低权限进程向高权限进程发送输入。

### 2. SendInput 权限限制
`SendInput` API 在非前台线程或跨进程时可能被系统拦截。

### 3. 需要管理员权限
跨进程输入可能需要以管理员身份运行。

### 4. 输入法状态影响
中文输入法状态可能影响 Unicode 输入或键盘事件处理。

### 5. 焦点恢复不彻底
虽然日志显示窗口已激活，但输入焦点可能仍在 WordFlow。

## 方案 7：点击客户区 + WM_PASTE 消息（2025-02-12 新增）

### 用户反馈
- 光标不再闪烁，有"要输入的感觉"
- 点击 WordFlow 时光标不会自动聚焦到文字内容（改进成功）

### 问题分析
日志显示一切正常：
```
✓ 窗口成功恢复到前台: 新建文本文档.txt - Notepad
✓ 剪贴板已设置
✓ Ctrl+V 已发送
```

1. 记事本窗口虽然在前台，但文本编辑框没有获得焦点

```csharp
var focusedWnd = GetFocus();
if (focusedWnd != IntPtr.Zero)
```

## 方案 8：以管理员权限运行（2025-02-12 新增）

### 用户建议
> "把它变成 EXE 格式，然后用管理员权限打开，是否可行？"

### 分析
这是一个非常合理的猜测！Windows 确实有以下安全限制：
1. **UIPI (User Interface Privilege Isolation)**: 普通权限程序不能向高权限窗口发送消息
2. **SendInput 限制**: 非前台/非管理员程序的输入可能被系统拦截
3. **剪贴板操作**: 跨进程粘贴可能受权限影响

### 解决方案
添加应用程序清单文件 (`app.manifest`)，声明需要管理员权限：

```xml
<?xml version="1.0" encoding="utf-8"?>
<assembly manifestVersion="1.0" xmlns="urn:schemas-microsoft-com:asm.v1">
  <assemblyIdentity version="1.0.0.0" name="WordFlow.app"/>
  <trustInfo xmlns="urn:schemas-microsoft-com:asm.v2">
    <security>
      <requestedPrivileges>
        <requestedExecutionLevel level="requireAdministrator" uiAccess="false" />
      </requestedPrivileges>
    </security>
  </trustInfo>
</assembly>
```

### 效果
- 编译后运行 WordFlow.exe 时会弹出 UAC 提示
- 用户点击"是"后，程序以管理员权限运行
- 跨进程输入限制可能被解除

## 方案 10：逐字符直接输入（Typeless 风格）

### 用户反馈
用户正在使用 **Typeless**，效果很好。Typeless 的实现方式可能是**逐字符直接输入**，而不是剪贴板粘贴。

### Typeless 可能的实现方式
类似语音输入工具的常见实现：
1. **逐字符输入**：直接模拟键盘输入每个字符
2. **不使用剪贴板**：避免剪贴板被占用或权限问题
3. **模拟真实打字**：像用户自己在打字一样

### 解决方案
使用 InputSimulator 的 `TextEntry` 方法逐字符输入：

```csharp
private async Task TypeTextWithSimulator(InputSimulator simulator, string text)
{
    foreach (char c in text)
    {
        if (c > 127) // 中文字符
        {
            // 中文使用剪贴板方式输入单个字符
            Clipboard.SetText(c.ToString());
            simulator.Keyboard.ModifiedKeyStroke(VirtualKeyCode.CONTROL, VirtualKeyCode.VK_V);
        }
        else // 英文和符号
        {
            // 直接输入
            simulator.Keyboard.TextEntry(c.ToString());
        }
        await Task.Delay(20); // 每个字符间隔 20ms
    }
}
```

### 流程
1. 激活目标窗口
2. 点击窗口客户区确保焦点
3. **逐字符输入文字**（Typeless 方式）
4. 中文用剪贴板粘贴，英文直接输入

### 预期效果
- 像 Typeless 一样直接输入文字
- 不用依赖 Ctrl+V 粘贴
- 更自然的输入体验

### 结果
- ✅ **成功！** 文字成功输入到外部窗口（记事本）
- 但速度较慢，是逐字符输入的

## 方案 11：优化为整段一次性输入（Typeless 风格最终版）

### 用户反馈
> "这回终于成功了！只不过这个字是一点一点打进去的。Typeless 在最后会停顿一下然后字一下子全输进来。"

### 问题分析
之前的方案虽然成功了，但是是**逐字符慢慢输入**（像打字一样），用户希望像 **Typeless** 那样：
1. 先停顿一下（等待识别完成）
2. 然后**整段文字一次性输入**

### 优化方案
改为**整段复制到剪贴板，一次性粘贴**：

```csharp
// 整段文字一次性粘贴（更快）
Clipboard.SetText(text);
simulator.Keyboard.ModifiedKeyStroke(VirtualKeyCode.CONTROL, VirtualKeyCode.VK_V);
```

同时优化延迟：
- 激活窗口：300ms → 100ms
- 点击窗口：200ms → 50ms  
- 粘贴间隔：100ms → 50ms

### 最终代码流程
```
1. 激活目标窗口 (100ms)
2. 点击窗口客户区 (50ms)
3. 整段复制到剪贴板
4. 一次性 Ctrl+V 粘贴
5. 输入空格分隔
```

### 预期效果
- ✓ 文字一次性全部输入（不是逐字）
- ✓ 速度更快，体验像 Typeless
- ✓ 有"停顿一下然后全部出现"的感觉

**原因**：
- 从未被实际使用

#### 4. 移除 WM_PASTE 消息相关代码

```diff
- WM_PASTE, WM_KEYDOWN, WM_KEYUP 等常量
- SendMessage, PostMessage, GetFocus, SetFocus API
```
**原因**：未实际使用，使用 InputSimulator 更简单。

#### 5. 移除 TypeTextWithSimulator 方法

**原因**：逻辑已合并到 SendTextToExternalWindowAsync。

### 最终精简后的流程
